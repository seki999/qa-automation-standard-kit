# Test Automation Guideline

## 目的

本ガイドラインは、社内プロジェクトへテスト自動化を横断導入する際の判断基準、責務分担、実装ルール、CI運用方針を標準化するための資料です。

## 自動化対象の選定基準

- リリースごとに繰り返し確認する回帰テスト
- 仕様変更の影響範囲が広い重要機能
- 手動確認に時間がかかるAPI契約、集計、状態遷移
- 障害発生時の影響が大きい業務フロー
- テストデータを安定して準備できるケース
- CIで短時間に実行でき、失敗原因を追跡できるケース

## 自動化しない方がよいテスト

- 仕様が頻繁に変わり、期待値が安定していない画面
- 目視の印象評価や探索的テストが主目的のケース
- 外部環境や日時に強く依存し、制御できないケース
- 実行頻度が低く、自動化コストを回収できないケース
- 一度限りの移行確認やデータ補正確認

## テストピラミッドの考え方

Unit Testを最も多く、Integration / API Testを中間、E2E Testを最小限にします。E2Eでしか確認できないユーザー体験は残しつつ、ロジックや契約は下位レイヤで高速に検証します。

## Unit / Integration / API / E2E の責務分担

| レベル | 責務 | 例 |
| --- | --- | --- |
| Unit | 業務ロジック、分岐、例外、計算 | Serviceの入力検証、ダッシュボード集計 |
| Integration | DB、Controller、Validation、JSON契約 | Repository検索、MockMvcによるHTTP検証 |
| API | サービス間契約、主要業務フロー | プロジェクト登録からダッシュボード反映まで |
| E2E | 実ユーザーに近い操作確認 | 画面からプロジェクト作成、結果登録 |

## 命名規則

- Unit Test: `対象クラス名Test`
- Integration Test: `対象機能IntegrationTest`
- Karate Feature: `業務領域-api.feature`
- Playwright Spec: `業務フロー.spec.ts`
- テスト名は「期待する振る舞い」を表す。例: `create_updatesTestCaseStatusAndStoresResult`

## テストデータ管理方針

- テスト内で必要なデータを作成し、固定IDに依存しない。
- API/E2Eは一意な名前にタイムスタンプを付け、並列実行の衝突を避ける。
- 初期サンプルデータはデモと手動確認用に限定する。
- 本番相当データを使う場合はマスキングし、個人情報を含めない。

## CIでの実行方針

- PRではUnit / Integration / Frontend build / API smoke / E2E smokeを実行する。
- E2Eの長時間シナリオは夜間またはリリース前ジョブに分離する。
- レポート、スクリーンショット、トレースを成果物として保存する。
- 失敗したテストを無効化する場合は、Issueと復旧期限を必ず紐づける。

## flaky test 対策

- 固定sleepではなく、要素・API応答・状態変化を待つ。
- テスト間で共有状態を持たない。
- 外部APIは契約テストまたはMockに置き換える。
- 失敗ログ、スクリーンショット、トレースを確認して原因分類する。
- リトライは暫定対策とし、根本原因を別途修正する。

## レビュー観点

- テスト対象の責務が適切なレイヤに置かれているか。
- テスト名から仕様が読めるか。
- 固定ID、固定日時、実行順依存がないか。
- CIで再現可能なデータ準備になっているか。
- 失敗時に原因を追えるアサーションとログがあるか。
- E2Eで過剰に細かいUI実装へ依存していないか。
