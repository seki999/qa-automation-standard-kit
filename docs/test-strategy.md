# Test Strategy

## テスト目的

プロジェクト管理・品質管理機能に対し、リリース判断に必要な信頼性を自動テストで継続的に担保します。単体ロジック、API契約、画面操作、CI実行をつなげ、変更時の回帰検知を早めます。

## テスト範囲

- プロジェクトの登録、一覧、更新、削除
- テストケースの登録、一覧、更新、削除
- テスト実行結果の登録
- ステータス更新
- ダッシュボード集計
- React画面からの主要操作

## テストレベル

| レベル | 対象 | 実行タイミング |
| --- | --- | --- |
| Unit | Service | 開発中、PR |
| Integration | Repository, Controller | PR |
| API | REST API主要シナリオ | PR、リリース前 |
| E2E | 画面主要シナリオ | PR smoke、夜間回帰 |

## テスト観点

- 正常系: 登録、取得、集計が期待通りに動くこと
- 異常系: 存在しないIDやValidationエラーが適切に返ること
- 境界: テストケース0件時の成功率が0になること
- 状態遷移: テスト結果登録によりテストケースステータスが更新されること
- 契約: APIのレスポンス構造がUIとテストに対して安定していること
- UX: 画面から主要操作が完了できること

## 品質基準

- PR時にCIが成功していること
- 重要なService分岐にUnit Testがあること
- Controllerの主要エンドポイントにIntegration Testがあること
- APIの主要業務フローがKarateで検証されていること
- E2E smokeがユーザー価値の高いシナリオをカバーしていること

## 自動化対象

- CRUDの正常系
- ダッシュボード集計
- API契約確認
- UIからのプロジェクト作成、テストケース作成、結果登録
- CIで短時間に実行できる回帰テスト

## 手動テスト対象

- UIの細かな見た目、文言、操作感
- 新規機能の探索的テスト
- 権限設計や例外フローの初回確認
- ブラウザや端末固有の視覚確認

## リスクと対策

| リスク | 対策 |
| --- | --- |
| E2Eが不安定になる | Playwrightを主軸にし、待機条件とテストデータを標準化する |
| テスト実行時間が長くなる | Unit/API中心に寄せ、E2Eはsmokeへ絞る |
| テストが仕様変更に追随しない | PRレビューでテスト更新観点を必須化する |
| テストデータが競合する | テスト内作成、タイムスタンプ、一意キーを使う |
| レポートが見られない | CI artifactとしてSurefire/Karate/Playwrightレポートを保存する |
